GitHub

Repositorio de código fuente. Streamlit Cloud lee directamente de aquí. No sube credenciales — el .gitignore excluye .env y secrets.toml.

Streamlit Cloud

Ejecuta el dashboard Python en sus servidores. Recibe las credenciales de BD mediante Secrets (configurados en la UI, nunca en el código).

Supabase

Base de datos PostgreSQL en la nube. Acepta conexiones remotas desde Streamlit Cloud vía el connection pooler en el puerto 6543.

🔑 ¿Por qué se usa el Connection Pooler de Supabase?

El host directo de Supabase (db.xxx.supabase.co:5432) expone solo IPv6. Streamlit Cloud utiliza IPv4 exclusivamente. El pooler de transacciones (aws-1-us-east-2.pooler.supabase.com:6543) resuelve esto al exponer un endpoint IPv4, además de gestionar el pool de conexiones automáticamente — ideal para entornos serverless.

Fase 2: Configurar Supabase

Supabase provee PostgreSQL gestionado en la nube con capa gratuita generosa (500 MB de base de datos, conexiones ilimitadas vía pooler).

1Crear Proyecto en Supabase

  1. Ve a supabase.com → Start your project
  2. Inicia sesión con GitHub (recomendado) o con email
  3. Clic en New Project
  4. Completa el formulario:
    • Name: etlweatherstack
    • Database Password: elige una contraseña segura y guárdala — la necesitarás después
    • Region: East US (North Virginia) — más cercana a los servidores de Streamlit Cloud en AWS
  5. Clic en Create new project — espera ~2 minutos

2Obtener las Credenciales de Conexión

Una vez creado el proyecto, ve a Project Settings → Database:

  1. En el panel izquierdo, clic en el ícono de engranaje ⚙️ → Database
  2. Desplázate hacia abajo hasta la sección Connection pooling
  3. Asegúrate de seleccionar Mode: Transaction
  4. Copia los valores — los necesitarás para el secrets.toml:
Campo en Supabase Variable en secrets.toml Ejemplo de valor
Host (pooler) DB_HOST aws-1-us-east-2.pooler.supabase.com
Port (pooler) DB_PORT 6543
User (pooler) DB_USER postgres.abcdefghijklmn
Password DB_PASSWORD tu_contraseña_segura
Database DB_NAME postgres

⚠️ Contraseña con caracteres especiales

Si tu contraseña contiene @, #, % u otros caracteres especiales, el código usa quote_plus() de urllib.parse para codificarlos correctamente en la URL de conexión. Esto ya está implementado en scripts/database.py. No necesitas modificar la contraseña.

3Crear las Tablas vía SQL Editor de Supabase

En el panel de Supabase, ve a SQL Editor (ícono de terminal >_ en la barra lateral). Crea un New query y ejecuta el siguiente SQL para crear las 3 tablas del proyecto:

-- ══════════════════════════════════════════════════════════════
-- TABLA 1: ciudades
-- ══════════════════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS ciudades (
    id          SERIAL PRIMARY KEY,
    nombre      VARCHAR(100) NOT NULL UNIQUE,
    pais        VARCHAR(100) NOT NULL,
    latitud     DOUBLE PRECISION,
    longitud    DOUBLE PRECISION,
    fecha_creacion TIMESTAMP DEFAULT NOW(),
    activa      BOOLEAN DEFAULT TRUE
);

CREATE INDEX IF NOT EXISTS ix_ciudades_nombre ON ciudades (nombre);

-- ══════════════════════════════════════════════════════════════
-- TABLA 2: registros_clima
-- ══════════════════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS registros_clima (
    id               SERIAL PRIMARY KEY,
    ciudad_id        INTEGER NOT NULL REFERENCES ciudades(id),
    temperatura      DOUBLE PRECISION NOT NULL,
    sensacion_termica DOUBLE PRECISION,
    humedad          DOUBLE PRECISION NOT NULL,
    velocidad_viento DOUBLE PRECISION NOT NULL,
    descripcion      VARCHAR(255) NOT NULL,
    codigo_tiempo    INTEGER,
    fecha_extraccion TIMESTAMP DEFAULT NOW(),
    fecha_creacion   TIMESTAMP DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS ix_registros_clima_ciudad_id
    ON registros_clima (ciudad_id);
CREATE INDEX IF NOT EXISTS ix_registros_clima_fecha_extraccion
    ON registros_clima (fecha_extraccion);
CREATE INDEX IF NOT EXISTS idx_ciudad_fecha
    ON registros_clima (ciudad_id, fecha_extraccion);

-- ══════════════════════════════════════════════════════════════
-- TABLA 3: metricas_etl
-- ══════════════════════════════════════════════════════════════
CREATE TABLE IF NOT EXISTS metricas_etl (
    id                       SERIAL PRIMARY KEY,
    fecha_ejecucion          TIMESTAMP DEFAULT NOW(),
    registros_extraidos      INTEGER NOT NULL,
    registros_guardados      INTEGER NOT NULL,
    registros_fallidos       INTEGER DEFAULT 0,
    tiempo_ejecucion_segundos DOUBLE PRECISION NOT NULL,
    estado                   VARCHAR(50) NOT NULL,
    mensaje                  VARCHAR(500)
);

CREATE INDEX IF NOT EXISTS ix_metricas_etl_fecha
    ON metricas_etl (fecha_ejecucion);

Clic en Run (o Ctrl+Enter). Deberías ver "Success. No rows returned" — las tablas se crearon correctamente.

📌 ¿Por qué crear las tablas por SQL y no desde Python?

Aunque SQLAlchemy puede crear tablas con Base.metadata.create_all(), hacerlo directamente en el SQL Editor de Supabase tiene ventajas:

  • Verificas visualmente la estructura antes de cargar datos
  • No necesitas configurar la conexión local para crear tablas
  • Puedes inspeccionar el esquema en Table Editor inmediatamente
  • Es más didáctico — ves el SQL real que genera el ORM

Streamlit Cloud necesita tres archivos clave para funcionar correctamente: requirements.txt, .streamlit/config.toml y el archivo Python del dashboard.

📌 Por qué versiones exactas

Usar >= puede instalar versiones más nuevas con cambios incompatibles. Con == garantizas que lo que funciona en local funciona en producción. Obtienes las versiones exactas con:

pip freeze | grep -E "^(streamlit|plotly|pandas|sqlalchemy|psycopg2|requests)="

2Verificar .streamlit/config.toml

Este archivo configura el comportamiento de Streamlit en producción. Debe existir en la raíz del proyecto:

[server]
headless = true
port = 8501
enableCORS = false

[theme]

backgroundColor = "#ffffff"
secondaryBackgroundColor = "#f0f2f6"
textColor = "#2c3e50"

El parámetro headless = true es obligatorio en producción — indica que no hay navegador local disponible para abrir automáticamente.

3Verificar scripts/database.py — Lectura de Secrets

El módulo de conexión debe leer credenciales desde st.secrets (Streamlit Cloud) y caer a .env solo en local. La lógica correcta:

def _get_db_config():
    # Intento 1: st.secrets (disponible en Streamlit Cloud)
    try:
        import streamlit as st
        host = st.secrets.get("DB_HOST", "")
        if host and host != "localhost":
            return {
                "host":     host,
                "port":     st.secrets.get("DB_PORT", "5432"),
                "user":     st.secrets.get("DB_USER", "postgres"),
                "password": st.secrets.get("DB_PASSWORD", ""),
                "dbname":   st.secrets.get("DB_NAME", "postgres"),
            }
    except Exception:
        pass  # st no disponible (scripts CLI locales)

    # Intento 2: variables de entorno / .env (desarrollo local)
    host = os.getenv("DB_HOST", "localhost")
    return {
        "host":     host,
        "port":     os.getenv("DB_PORT", "5432"),
        "user":     os.getenv("DB_USER", "postgres"),
        "password": os.getenv("DB_PASSWORD", ""),
        "dbname":   os.getenv("DB_NAME", "weatherstack_etl"),
    }

⚠️ Error común: fallback silencioso a localhost

Si usas st.secrets["DB_HOST"] con corchetes y la clave no existe, lanza KeyError. El except Exception lo captura silenciosamente y cae a localhost, generando el error: "Connection refused to 127.0.0.1:5432". Siempre usa st.secrets.get("DB_HOST", "") y verifica que el valor no sea localhost.

3Configurar Python 3.11 — Directamente en Streamlit Cloud

Streamlit Cloud usa Python 3.9 por defecto, pero nuestro proyecto necesita Python 3.11 porque es significativamente más rápido (hasta 60% en benchmarks de CPython) y compatible con todas nuestras dependencias. Este cambio se realiza directamente en la interfaz de Streamlit Cloud al momento del despliegue:

  1. En la pantalla de "Deploy an app", antes de hacer clic en Deploy, haz clic en "Advanced settings"
  2. En el campo "Python version", cambia el valor de 3.9 a 3.11
  3. Streamlit Cloud instalará Python 3.11 en el contenedor de la app automáticamente

📌 ¿Por qué Python 3.11 y no 3.9 o 3.12?

  • Rendimiento: CPython 3.11 es ~10-60% más rápido que 3.9/3.10 gracias al proyecto "Faster CPython"
  • Pandas 2.x: pandas 2.3 requiere Python 3.9+ y aprovecha mejor Python 3.11
  • SQLAlchemy 2.0: funciona óptimamente en 3.11
  • Estabilidad: Python 3.12+ puede tener incompatibilidades con algunas dependencias; 3.11 es la opción más estable y probada para producción en Streamlit Cloud

⚠️ Nota importante

No es necesario crear un archivo .python-version en el repositorio — la configuración de Python 3.11 en Advanced settings de Streamlit Cloud es suficiente y queda guardada para todos los redespliegues futuros de la app.

  2. Clic en el menú ⋮ (tres puntos verticales) de la app
  3. Selecciona Settings
  4. Ve a la pestaña Secrets
  5. Pega el mismo bloque de secrets
  6. Clic en Save — la app se reinicia automáticamente

⚠️ Formato TOML obligatorio

Los secrets deben estar en formato TOML (igual que el archivo secrets.toml). Las claves son strings, van entre comillas dobles. Usar comillas simples o formato JSON causará errores de parseo.

✓ Correcto: DB_PORT = "6543"
✗ Incorrecto: DB_PORT = 6543 (sin comillas)

Fase 7: Errores Comunes y Soluciones

Referencia rápida de los errores más frecuentes en despliegues de Streamlit Cloud con Supabase y cómo resolverlos.

🔴 Error 1 — Connection refused a localhost:5432

OperationalError: connection to server at "localhost" (127.0.0.1), port 5432 failed:
Connection refused

Causa: Los secrets no están configurados, están vacíos, o DB_HOST tiene el valor localhost.

Solución:

  1. Settings → Secrets → verifica que DB_HOST apunta al pooler de Supabase
  2. Verifica que el código usa st.secrets.get() en lugar de st.secrets[]
  3. El código debe verificar: if host and host != "localhost":

🔴 Error 2 — ModuleNotFoundError: No module named 'scripts'

ModuleNotFoundError: No module named 'scripts'

Causa: Falta el archivo scripts/__init__.py en el repositorio.

Solución:

# Crea el __init__.py vacío

🔴 Error 3 — SSL connection required

OperationalError: SSL connection has been closed unexpectedly
# o
SSL SYSCALL error: EOF detected

Causa: Supabase requiere SSL obligatorio. La URL de conexión no incluye ?sslmode=require.

Solución: Verifica que database.py construye la URL así:

DATABASE_URL = (
    f"postgresql://{quote_plus(DB_USER)}:{quote_plus(DB_PASSWORD)}"
    f"@{DB_HOST}:{DB_PORT}/{DB_NAME}?sslmode=require"
)

🔴 Error 4 — Contraseña con @ rompe la URL

OperationalError: invalid URL
# o conexión a host incorrecto por el @ en la contraseña

Causa: El carácter @ en la contraseña rompe el parsing de la URL de conexión de PostgreSQL.

Solución: Siempre usa quote_plus para user y password:

from urllib.parse import quote_plus

DATABASE_URL = (
    f"postgresql://{quote_plus(DB_USER)}:{quote_plus(DB_PASSWORD)}"
    f"@{DB_HOST}:{DB_PORT}/{DB_NAME}?sslmode=require"
)

🔴 Error 5 — App se despliega pero queda en "Connecting..."

# La app carga el spinner pero nunca muestra datos

Causa: El host de Supabase configurado es el directo (db.xxx.supabase.co:5432), que solo responde por IPv6. Streamlit Cloud solo tiene IPv4.

Solución: Usa el connection pooler (transaction mode, puerto 6543):

# ✗ Host directo (solo IPv6 — NO funciona en Streamlit Cloud)
DB_HOST = "db.abcdefgh.supabase.co"
DB_PORT = "5432"

# ✓ Pooler de transacciones (IPv4 — SÍ funciona)
DB_HOST = "aws-1-us-east-2.pooler.supabase.com"
DB_PORT = "6543"
DB_USER = "postgres.tu_project_ref"  # nota el prefijo postgres.

🟡 Error 6 — Versión de dependencia incompatible

ERROR: Could not find a version that satisfies the requirement pandas==3.0.0

Causa: Python 3.9 (versión por defecto de Streamlit Cloud) no soporta versiones recientes de pandas/numpy.

Solución: Configura Python 3.11 en Advanced settings (como se describe en la Fase 5, paso 3) y usa versiones compatibles en requirements.txt.

🟡 Error 7 — Too many connections (Supabase)

OperationalError: FATAL: too many connections for role "postgres.xxx"

Causa: En el plan gratuito de Supabase, el pooler en modo session limita conexiones simultáneas.

Solución: Usa el pooler en modo Transaction (puerto 6543). Agrega también pool_pre_ping=True al engine para reciclar conexiones muertas:

engine = create_engine(
    DATABASE_URL,
    echo=False,
    pool_pre_ping=True,
    connect_args=connect_args
)

Resumen del Despliegue

Completaste el despliegue completo de un pipeline ETL con dashboard en la nube. Esta es la arquitectura final:

☁️ Streamlit Cloud

  • Dashboard accesible públicamente
  • Python 3.11 para rendimiento óptimo
  • Redeploy automático con cada git push
  • Secrets gestionados de forma segura

🗄️ Supabase

  • Pooler Transaction (IPv4, puerto 6543)

  • Credenciales solo en Streamlit Secrets
  • quote_plus para contraseñas especiales
  • sslmode=require en la conexión
